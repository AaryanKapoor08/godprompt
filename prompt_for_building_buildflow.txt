# STEP 3 — WRITE `codex/buildflow.md`
Create or replace `codex/buildflow.md`. Match this structure exactly.

## File structure (top to bottom)

1. **Title + 1–3 line framing.** Practical, execution-focused. State that this is a stabilization/re-architecture effort, not a greenfield build.

2. **Prerequisites.** Only what the current repo and current plan require. Drop stale architecture assumptions. Include: Node/npm versions in use, current test command, where production code lives (`extension/src/`), where tests live (`extension/test/`), how to run the regression corpus once introduced.

3. **Global rules.** Flat list of execution rules that apply across every phase:
   - no-regression rollout (corpus must stay green)
   - Gemma freeze (no edits to Gemma prompts, cleanup, or retry)
   - branch separation (`LLM branch` and `Text branch` evolve independently)
   - provider separation (Google and OpenRouter evolve independently)
   - token-budget enforcement (every prompt change asserts measured tokens against the cap)
   - checkpoint-first phase completion (no phase advances until every `[ ]` is `[x]` with proof)
   - no destructive refactors (existing files stay until their last responsibility moves)
   - acceptance gates before broad rollout (eval gate before any default-model change)

4. **Phases.** One section per phase. Use this exact per-phase template:

   ### Phase N — <short title>
   - **Goal.** One sentence.
   - **Why now.** One or two sentences explaining the dependency on prior phases.
   - **Tasks.** Numbered, concrete, verb-led. No vague tasks.
   - **Code areas.** Files or directories likely touched (e.g. `extension/src/lib/rewrite-core/validate.ts`).
   - **Checkpoints.** Markdown checklist with `[ ]`. Each checkpoint must be independently verifiable. Example:
     - [ ] Regression corpus runs in CI with non-zero entries
     - [ ] `LLM branch` first-pass token count measured and logged below `1000`
   - **Risks.** What could regress, what could leak past validation.
   - **Commit message.** Suggested message in conventional-commit style matching the repo (e.g. `feat(rewrite-core): add constraint extractor`). Include scope.
   - **Rollback / containment.** Only if the phase touches user-facing behavior. Note the safe revert path.

5. **Phase order (recommended).** Use this order unless reading the files reveals a safer alternative. If you change it, add a short paragraph in the file explaining why.
   1. Lock acceptance criteria and regression corpus
   2. Token-budget measurement seam (before any prompt edits)
   3. Shared `rewrite-core` primitives (`types`, `normalize`, `constraints`, `validate`, `repair`, `fallback`)
   4. `LLM branch` migration to compact-contract prompting
   5. `Text branch` hardening within the narrow allowed scope
   6. Google-specific policy alignment (model registry, request policy, retry policy, budget policy)
   7. OpenRouter live-catalog + curated-chain + cooldown + account-status
   8. UI alignment (popup model list reflects live curated chain; visible order = runtime order)
   9. Verification matrix + rollout gates

6. **Testing strategy.** Short section forcing every phase to demonstrate the relevant subset of:
   - regression corpus checks (per-branch, per-provider; Gemma exempt)
   - token-budget checks (first pass and retry, both branches)
   - provider-specific verification (Google quota path, OpenRouter cooldown/cycle, account-status bucket detection)
   - branch-specific verification (LLM staged workflows, Text deliverable preservation)
   - no-output-answering regressions
   - deliverable-preservation regressions
   - no-markdown / no-brief / no-duplicate-summary regressions
   - manual verification matrix at the right phases (after Phase 4, after Phase 7, after Phase 9), not only at the end

7. **Token-budget enforcement (first-class).** Short section pointing to where in the phases the following are introduced:
   - measurement seam (Phase 2)
   - separate budgets for first pass vs retry (Phase 4 for LLM, Phase 5 for Text)
   - branch-specific enforcement (Phase 4, Phase 5)
   - production prompts ship without examples; debug-mode prompts may include them (Phase 4 build-flag work)

8. **Final section.** Headed `Phase Strategy Summary`. Four lines, one each:
   - **Highest-risk phase:** name + one-line reason.
   - **Highest-leverage phase:** name + one-line reason.
   - **Start here:** the first phase to actually implement.
   - **Do not touch early:** what should be deferred and why.
